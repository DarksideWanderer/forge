# Forge 开发规范

本文档规定了 Forge 包管理器核心代码的开发标准、架构设计及协作流程。

## 1. 技术栈要求

- **编程语言**: C++20 (强制要求)
- **构建系统**: CMake (>= 3.20)
- **编译器支持**: 优先支持 Clang，兼顾 GCC
- **外部依赖**: 尽量减少第三方库，核心逻辑优先使用标准库
- **跨平台支持**: 目标平台包括 Linux、macOS 和 Windows

## 2. 核心架构设计

Forge 采用模块化设计，主要分为以下层次：

### 2.1 配置解析层 (`Config`) 
- 负责 `forge.toml` 的反序列化。
- 校验字段合法性，提供全局配置单例。

### 2.2 依赖图层 (`Dependency`)
- 实现依赖树/图的解析算法（Version SAT 求解或简单的深度优先搜索）。
- 处理版本冲突检测。

### 2.3 下载与缓存层 (`Fetch`)
- 负责从指定源（Git/HTTP）获取源代码或二进制包。
- 维护本地缓存目录 `~/.forge/cache`。

### 2.4 构建生成层 (`Generator`)
- 根据 `forge.toml` 自动生成透明的构建配置（如 `CMakeLists.txt`）。
- 确保依赖库的路径和编译参数正确传递。

### 2.5 命令行接口 (`CLI`)
- 使用统一的命令风格（如 `forge build`, `forge add`）。

## 3. 代码风格

### 3.1 基础风格
- 遵循类似 **LLVM** 的编码风格。
- 使用驼峰命名法（`PascalCase`）命名类，小驼峰（`camelCase`）命名变量和函数。

### 3.2 错误处理
- 禁止使用 C 风格错误码，优先使用 `std::expected` (C++23) 的替代方案或异常处理。
- 内部逻辑必须保证异常安全性。

### 3.3 文档与注释
- 所有公开 API 必须包含 Doxygen 注释。
  [参考](https://blog.csdn.net/qq_41204464/article/details/102458103)
  
- 关键算法（如依赖解析）需在代码中注明设计思路。

## 4. 测试驱动

- **单元测试**: 每个核心类必须有对应的单元测试（位于 `tests/` 目录）。
- **集成测试**: 每个功能点（如 `forge init`）需通过脚本进行端到端测试。

## 5. 开发流程

1. **最小版本目标**: 初期仅支持 `Clang` + `CMake` + `无模块化` 项目。
2. **提交规范**: 提交信息需遵循 [Conventional Commits](https://www.conventionalcommits.org/) 规范（例如 `feat: add toml parser`）。
3. **PR 流程**: 所有更改必须经过 CI 和至少一名成员 Code Review。

## 6. 项目结构

Forge 仓库的规范结构如下：

```text
forge/
├── CMakeLists.txt          # 项目主构建文件
├── Readme.md               # 项目介绍
├── LICENSE                 # 开源协议
├── TODO/                   # 路线图与计划计划
├── docs/                   # 项目文档与规范
│   ├── 开发规范.md
│   └── 包管理器配置规范.md
├── src/                    # 源代码
│   ├── main.cpp            # 程序入口
│   ├── cli/                # 命令行交互逻辑
│   ├── config/             # TOML 解析与配置管理
│   ├── dependency/         # 依赖树图算法
│   ├── fetch/              # 下载与缓存管理
│   ├── generator/          # 构建文件生成器 (CMake/Meson)
│   └── utils/              # 通用工具函数 (文件/网络/字符)
├── include/                # 公开头文件 (如有)
├── tests/                  # 测试用例
│   ├── unit/               # 单元测试
│   └── integration/        # 集成测试
├── example/                # 示例项目
└── external/               # 第三方库 (Git Submodules 或 源码)
```
